## S-01. 로그인 및 역할별 초기 화면 진입

**목적:** 각 역할 계정으로 로그인 시 올바른 사이드바 섹션이 펼쳐지고, 기본 랜딩 패널이 정확히 표시되는지 확인한다.

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-01-01 | User 계정 로그인 | user 계정 ID·PW 입력 후 로그인 | 로그인 오버레이 닫힘, 사이드바 MY 섹션 자동 펼침, `user-status` 패널 표시 |
| TC-01-02 | Developer 계정 로그인 | developer 계정 ID·PW 입력 후 로그인 | DEVELOPER 섹션 자동 펼침, `developer` 패널(내 현황) 표시 |
| TC-01-03 | Admin 계정 로그인 | admin 계정 ID·PW 입력 후 로그인 | ADMIN 섹션 자동 펼침, `approve`(등록 승인) 패널 표시 |
| TC-01-04 | 잘못된 비밀번호 입력 | 임의 ID + 틀린 PW | 로그인 오버레이 유지, 오류 메시지(`.login-error`) 표시 |
| TC-01-05 | 로그아웃 | 로그인 후 사이드바 하단 `로그아웃` 버튼 클릭 | 로그인 오버레이 재표시, 계정 정보 초기화(이름·이메일·역할 배지 `—`로 복귀) |
| TC-01-06 | Topbar 역할 칩 표시 | 각 역할 로그인 | Topbar 우측 `.user-chip` 내 역할명(User / Developer / Admin) 정확히 표시 |

## S-02. 마켓플레이스 탐색 및 Agent 구독

**목적:** User가 마켓플레이스에서 Agent를 검색·필터링하고 구독하면 내 현황에 반영되는지 검증한다.

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-02-01 | 키워드 검색 | 검색창에 `법무` 입력 후 검색 버튼 클릭 | 법무·계약 관련 Agent 카드만 노출 |
| TC-02-02 | 카테고리 필터 선택 | `코드 리뷰` 칩 클릭 | 해당 카테고리 Agent 카드만 표시, 칩 활성 스타일(`.on`) 적용 |
| TC-02-03 | 전체 필터로 복귀 | `전체` 칩 클릭 | 모든 Agent 카드 재표시 |
| TC-02-04 | Agent 구독 | `법무 계약 검토 Agent` 카드의 `구독` 버튼 클릭 | 버튼이 `구독 중` 또는 구독 완료 상태로 변경, 카드에 구독 배지 표시 |
| TC-02-05 | 구독 후 내 현황 반영 | 구독 완료 후 `내 현황` 메뉴 이동 | 구독 중인 Agent 수(stat-card) +1 증가, 해당 Agent 카드(`sub-agent-card`) 목록에 추가 |
| TC-02-06 | A2A 지원 칩 표시 확인 | 마켓플레이스 카드 목록 확인 | A2A 허용 설정된 Agent 카드에만 `.a2a-chip` 표시 |
| TC-02-07 | 검색 결과 없음 | 존재하지 않는 키워드 입력 | 빈 상태(`empty-state`) 또는 "결과 없음" 메시지 표시 |

## S-03. 구독 취소 및 재구독

**목적:** 구독 취소 시 내 현황에서 제거되고, 재구독 시 다시 복원되는지 확인한다.

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-03-01 | 구독 취소 | `내 현황` → 구독 Agent 카드의 `구독 취소` 버튼 클릭 | 카드가 목록에서 제거, 구독 중 Agent 수 감소 |
| TC-03-02 | 취소 후 마켓플레이스 버튼 상태 | 취소한 Agent가 마켓플레이스로 이동 | 해당 카드의 구독 버튼이 `구독` (미구독 상태)으로 표시 |
| TC-03-03 | 재구독 | 취소한 Agent 재구독 | 내 현황에 다시 표시, 이전 사용 데이터(쿼리 수 등) 유지 또는 초기화 확인 |
| TC-03-04 | 구독 0건 상태 | 모든 구독 취소 후 내 현황 확인 | "더 많은 Agent를 구독해보세요" 유도 영역 표시, MARKETPLACE 바로가기 버튼 동작 |

## S-04. 문서 업로드 → S3 저장 확인

**목적:** User가 문서를 업로드하면 S3에 저장되고, 내 문서 현황에 `S3 완료 · 승인 대기` 상태로 등록되는지 검증한다.

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-04-01 | Agent 선택 시 Collection 자동 표시 | `문서 업로드` 패널 → Agent 선택 드롭다운에서 `법무 계약 검토 Agent` 선택 | `collectionDisplay` 필드에 `legal_contracts_v2` 자동 입력 |
| TC-04-02 | 파일 선택(클릭) | 파일 선택 영역 클릭 → `.pdf` 파일 선택 | 파일 목록(`.file-list`)에 파일명·크기 표시 |
| TC-04-03 | 파일 드래그 앤 드롭 | `.drop-zone`에 `.docx` 파일 드래그 후 드롭 | 드래그 오버 스타일 적용 후 파일 목록 추가 |
| TC-04-04 | 파일 삭제 | 추가된 파일의 `✕` 버튼 클릭 | 해당 파일 항목이 목록에서 제거 |
| TC-04-05 | 지원하지 않는 형식 업로드 | `.exe` 파일 선택 시도 | 파일 선택 차단 또는 오류 메시지 표시 |
| TC-04-06 | 필수 항목 미입력 후 제출 | Agent 미선택 상태에서 `업로드 요청` 클릭 | 폼 유효성 오류 표시, 업로드 미진행 |
| TC-04-07 | JWT Token 표시/숨김 토글 | JWT 입력창 옆 `SHOW` 클릭 | `password` → `text` 타입으로 전환, `HIDE` 텍스트로 변경 |
| TC-04-08 | Request Body JSON 유효성 검사 | 잘못된 JSON 입력 후 업로드 요청 | 파싱 오류 메시지 표시, 제출 차단 |
| TC-04-09 | 정상 업로드 시뮬레이션 | 모든 필드 입력 후 `업로드 요청` 클릭 | 업로드 오버레이(`.upload-overlay`) 표시 → 완료 후 `내 문서 현황`에 `S3 완료·승인 대기` 상태로 신규 행 추가 |
| TC-04-10 | 초기화 버튼 | 입력 완료 후 `초기화` 클릭 | 모든 필드(Agent 선택, 파일 목록, DAG, JWT, Request Body) 초기 상태로 복원 |

## S-05. Developer 문서 승인 → VectorDB 등록 완료 E2E

**목적:** User 업로드 → Developer 승인 → Airflow 임베딩 → VectorDB 등록까지 전체 파이프라인이 순서대로 완료되는지 검증한다.

| TC | 테스트 케이스 | 입력 / 조건 | 기대 결과 |
|----|-------------|------------|---------|
| TC-05-01 | Developer 문서 승인 패널 배지 카운트 | User가 문서 업로드 완료 후 Developer 로그인 | 사이드바 `문서 승인` 메뉴의 nav-badge 카운트 +1 |
| TC-05-02 | Agent 필터 탭 동작 | 문서 승인 패널 → `⚖️ legal-review-agent` 탭 클릭 | 해당 Agent 문서만 테이블에 표시, 다른 Agent 행 숨김 |
| TC-05-03 | Request Body 미리보기 | 대기 목록의 `{"collection_name":"doc_s...` 클릭 | 전체 JSON 내용이 팝업·툴팁·상세 뷰로 표시 |
| TC-05-04 | 문서 승인 처리 | 대기 목록의 `승인` 버튼 클릭 | 행이 대기 탭에서 제거, `처리 내역` 탭으로 이동, 상태 `승인 · 임베딩 중` 표시 |
| TC-05-05 | 대기 건수 감소 | 승인 후 | 승인 대기 뱃지(`pendingCount`) 및 Agent 카드 `대기` 카운트 -1 |
| TC-05-06 | 임베딩 완료 후 VectorDB 등록 | Airflow DAG 완료 시점(백엔드 이벤트) | 처리 내역 탭 상태가 `승인 · VectorDB 등록`으로 업데이트 |
| TC-05-07 | User 문서 현황 파이프라인 갱신 | Developer 승인 후 User 로그인 → 내 문서 현황 확인 | 해당 문서의 파이프라인 `S3 → 승인 → 임베딩 → VectorDB` 단계 순서대로 완료 표시 |
| TC-05-08 | 모든 대기 건 처리 후 빈 상태 | 대기 목록 전체 처리 | `emptyPending` 영역(✅ 대기 중인 문서가 없습니다) 표시 |

## S-06. Developer 문서 반려 → User 재업로드

**목적:** Developer가 문서를 반려하면 User 문서 현황에 반려 상태가 표시되고, 사용자가 재업로드할 수 있는지 확인한다.

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-06-01 | 문서 반려 처리 | Developer → 대기 목록 `반려` 버튼 클릭 | 행이 처리 내역으로 이동, 상태 `반려` 표시 |
| TC-06-02 | 반려 사유 기록 | 반려 시 사유 입력 모달 제공(또는 선택) | 처리 내역의 결과 컬럼에 반려 사유 표시 |
| TC-06-03 | User 문서 현황 반영 | User 로그인 → 내 문서 현황 확인 | 해당 파일 파이프라인이 `S3 → 반려(✕)` 상태로 표시 |
| TC-06-04 | 반려 건수 stat 카드 반영 | 내 문서 현황 통계 확인 | ❌ 반려 카운트 증가 |
| TC-06-05 | 반려 문서 재업로드 | 동일 파일 다시 업로드 요청 | 새로운 문서 행이 `S3 완료·승인 대기` 상태로 추가 |

## S-07. Developer Agent 등록 신청 → Admin 승인 → Gateway Key 발급 → Validation 통과 E2E

**목적:** Developer의 신규 Agent 등록 신청부터 Validation 통과 후 MARKETPLACE 공개까지 전체 플로우를 검증한다.

### Step 1 — 등록 신청 폼 작성 및 제출

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-07-01 | Agent 이름 중복 확인 | 이미 존재하는 이름 입력 후 `중복 확인` | 중복 오류 메시지 표시, 제출 차단 |
| TC-07-02 | Agent 이름 중복 없음 확인 | 신규 이름 입력 후 `중복 확인` | 사용 가능 메시지 표시 |
| TC-07-03 | 외부 URL 중복 확인 | 이미 사용 중인 Route Path 입력 후 `중복 확인` | 중복 오류 메시지 표시 |
| TC-07-04 | 필수 항목 누락 제출 | 필수(★) 항목 하나 비워두고 `신청 제출` | 해당 필드 오류 표시, 제출 차단 |
| TC-07-05 | 임시 저장 | 일부 입력 후 `임시 저장` | 입력 데이터가 유지되어 다음 접속 시 복원 |
| TC-07-06 | 정상 제출 | 모든 필수 항목 입력 후 `신청 제출` | Wizard Step 1 완료 → Step 2(검토 및 승인) 화면으로 전환, 진행 트래커 `신청 접수` 완료·`관리자 검토 중` 활성 표시 |
| TC-07-07 | 제출 후 Developer 내 현황 반영 | 신청 제출 완료 후 `내 현황` 이동 | 신청한 Agent가 목록에 `검토 중` 상태로 표시 |

### Step 2 — Admin 승인 및 Kong 토큰 발급

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-07-08 | Admin 승인 목록 배지 증가 | Developer 신청 완료 후 Admin 로그인 | 사이드바 `등록 승인` nav-badge 카운트 +1 |
| TC-07-09 | 승인 처리 | Admin → 등록 승인 패널 → `승인` 버튼 클릭 | 상태 `검토 중` → `승인 완료`로 변경, `Key 발급 →` 버튼 표시 |
| TC-07-10 | Gateway Key 발급 | `Key 발급 →` 클릭 또는 gwkey 패널에서 발급 | Kong 인증 토큰 생성, 개발자 이메일(Agent Owner)로 토큰 발송 |
| TC-07-11 | Developer Step 2 승인 완료 화면 전환 | 승인 + 토큰 발급 후 Developer 화면 새로고침 | Wizard Step 2 화면이 `승인 완료·Kong 인증 토큰 발급` 상태로 전환, 토큰 복사 버튼 활성 |
| TC-07-12 | 토큰 복사 버튼 | `복사` 버튼 클릭 | 버튼 텍스트 `복사됨 ✓`로 변경, 클립보드에 토큰 값 복사 |
| TC-07-13 | Pod 재시작 완료 요청 | `Pod 재시작 완료 · Validation 요청` 버튼 클릭 | Wizard Step 3(Validation 대기) 화면으로 전환 |

### Step 3 — Validation 실행 및 통과

| TC | 테스트 케이스 | 입력 | 기대 결과 |
|----|-------------|------|---------|
| TC-07-14 | Validation 모달 실행 | Admin → 개발자 내 현황 또는 별도 버튼 → `검증 시작` 클릭 | Validation 모달 표시, 진행 바 0% 시작 |
| TC-07-15 | Step 1 Route Health Check | 모달 실행 후 자동 진행 | `vs-running` → `vs-pass` 전환, 로그에 `401 Unauthorized (route 존재 확인)` 표시 |
| TC-07-16 | Step 2 Gateway Key 인증 | Step 1 통과 후 자동 진행 | `Authorization: Bearer → 200 OK` 로그, `vs-pass` 전환 |
| TC-07-17 | Step 3 A2A 호환성(허용 설정) | A2A 허용 Agent | A2A 스킬 목록 표시 및 A2A 호출 테스트 결과 `vs-pass` |
| TC-07-18 | Step 3 A2A Skip(미허용) | A2A 미허용 Agent | A2A 단계 `⏭ Skip` 표시 |
| TC-07-19 | Validation 전체 통과 | 3단계 완료 | 최종 결과 `.val-result.pass` 표시, Agent 상태 `ACTIVE`, MARKETPLACE에 카드 공개 |
| TC-07-20 | Validation 통과 후 내 현황 | Developer 내 현황 이동 | 해당 Agent 행의 Validation 뱃지가 `✅ 검증 완료`로 변경 |

## S-08. Validation 실패 → 재시도 플로우

| Milvus 동기화 | 문서 승인·임베딩 완료 후 해당 컬렉션의 벡터 수가 Admin Milvus 패널에 반영 |
