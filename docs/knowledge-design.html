S3 아키텍처 비교 — Case A vs Case B

문서가 어디에 저장되고 DAG이 어떻게 가져가는지의 차이입니다.

Case A — Developer 자체 S3

대안
👤
End-user 문서 업로드
doc-upload 패널에서 파일 선택
🖧
플랫폼 → Developer S3 전송
플랫폼이 Developer S3 버킷으로 직접 PUT
s3://dev-bucket/uploads/…
🧑‍💻
Developer 승인/반려 (doc-approve)
승인 → S3 유지 / 반려 → S3 삭제
⚙️
DAG 실행 (Developer S3 직접 접근)
자체 S3에서 파일 읽어 처리
🗄️
VectorDB(Milvus) 적재

✔ 장점

  • 플랫폼 스토리지 비용 없음
  • Developer 데이터 통제권

✖ 단점

  • Developer마다 S3 구성 필요
  • 플랫폼이 외부 S3 접근 키 관리
  • 보안·장애 책임 불명확

Case B — 플랫폼 S3 통합 관리

권장
👤
End-user 문서 업로드
doc-upload 패널에서 파일 선택
🖧
플랫폼 S3 임시 보관
플랫폼이 소유한 S3에 저장
platform-s3/agents/{id}/users/{uid}/file
🧑‍💻
Developer 승인/반려 (doc-approve)
반려 시 플랫폼 S3에서 파일 삭제
⚙️
플랫폼 백엔드 → DAG Trigger
DAG이 API 호출로 S3 경로 수신
GET /api/documents/{doc_id}/path
🗄️
VectorDB(Milvus) 적재 → 상태 콜백
DAG 완료 후 플랫폼으로 상태 업데이트
POST /api/documents/{doc_id}/status

✔ 장점

  • 보안·스토리지 일원화 관리
  • Developer S3 구성 불필요
  • 파일 생명주기 플랫폼 통제
  • DAG 표준 인터페이스 제공

✖ 단점

  • 플랫폼 S3 비용 발생
  • 백엔드 API 설계 선행 필요

화면(패널)별 변경 영향 범위

Knowledge 기능 도입 시 기존 ui-requirements.html에서 변경되는 패널입니다.

apply
Developer
  • Knowledge 사용 여부 토글 추가
  • DAG ID 입력 필드 (토글 ON 시 노출)
  • S3 경로 설정 — Case B는 플랫폼 자동
developer
Developer
  • Agent 상세에 Knowledge 설정 섹션
  • DAG ID 수정 기능
  • Knowledge 활성/비활성 토글
doc-approve
Developer
  • 기존 패널 유지 (핵심 화면)
  • 승인 시 DAG Trigger 연동
  • 반려 시 S3 삭제 트리거
doc-upload
User · Developer
  • Knowledge 미활성 Agent는 업로드 버튼 비활성
  • Agent별 Knowledge 지원 여부 뱃지 표시
doc-status
User · Developer
  • 처리 상태 컬럼 추가 (단계별)
  • Milvus 컬렉션명 표시 (등록 완료 시)
  • 반려 사유 노출
milvus
Admin
  • 컬렉션 목록에 연결 Agent명 컬럼
  • 등록 문서 수 표시
