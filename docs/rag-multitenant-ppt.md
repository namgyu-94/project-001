## [Slide 2] 문제 정의 — 왜 지금 필요한가

### 현재 상황
전사 단일 VectorDB에 여러 팀의 RAG 문서가 혼재

### 3가지 핵심 위험

| # | 문제 | 결과 |
|---|------|------|
| 1 | **기밀 노출** | 인사·재무 등 팀 내부 문서가 타 팀 검색 결과에 노출 |
| 2 | **필터 신뢰 불가** | 메타데이터 필터 한 번 누락 시 권한 경계 전체 붕괴 |
| 3 | **도입 장벽** | VectorDB·임베딩·보안 설정을 팀이 직접 구축해야 해 수 주 소요 |

> "보안팀 검토 통과 불가" → RAG 도입 자체가 중단되는 구조

## [Slide 3] 핵심 해결 원칙

### 우리의 접근: 애플리케이션 필터 → 인프라 레벨 격리

```
기존 방식 (취약)          제안 방식 (안전)
─────────────────         ─────────────────────
단일 Milvus Collection    팀별 전용 Collection
메타데이터 필터 의존  →   Collection 자체가 분리
필터 누락 = 전체 노출     필터 누락 = 영향 없음
```

### 3대 격리 원칙
1. **스토리지 격리** — 팀별 전용 MinIO Bucket (타 팀 접근 키 없음)
2. **파이프라인 격리** — 팀별 전용 Airflow DAG (다른 팀 데이터 처리 불가)
3. **VectorDB 격리** — 팀별 전용 Milvus Collection (검색 범위 물리적 분리)

## [Slide 5] 플랫폼 레이어 — 자원 할당 & 보안 인증

### Step 1: 자원 할당 (Resource Provisioning)

RAG 도입을 원하는 팀이 플랫폼에 신청하면 **전용 자원 세트**를 자동 생성

```
신청 → 플랫폼 승인 → 자동 프로비저닝
         ├── Airflow DAG (팀 전용)
         ├── MinIO Bucket (팀 전용)
         └── Milvus Collection (팀 전용)
```

### Step 2: 보안 인증 (Security Auth)

| 인증 수단 | 역할 |
|-----------|------|
| **JWT Token** | 문서 업로드 API 호출 시 팀 신원 검증 |
| **Gateway Key** | 플랫폼 내부 자원 접근 제어 |
| **RBAC** | 역할 기반 접근 — 본인 팀 자원만 조작 가능 |

> 인증 통과 후에만 해당 팀의 전용 자원에 접근 허용

## [Slide 6] 문서 업로드 파이프라인

### 전체 흐름 (팀 A 예시)

```
① 팀 A가 문서 파일 선택
        ↓
② JWT Token + Airflow DAG 선택 + Request Body 입력
        ↓
③ 플랫폼 보안 인증 (JWT 검증)
        ↓
④ MinIO Bucket-A 에 파일 저장
   (Bucket-B 접근 불가 — 키 없음)
        ↓
⑤ S3 이벤트 → Airflow DAG-A 자동 트리거
        ↓
⑥ 임베딩 모델 처리 (청킹 → 벡터화)
        ↓
⑦ Milvus Collection-A 에 벡터 저장
   (Collection-B 와 물리적으로 분리)
        ↓
⑧ RAG 검색 시 Collection-A 만 조회
```

### 핵심 포인트
- 팀 B가 팀 A의 Bucket-A에 접근하려면 **별도 AccessKey/SecretKey** 필요 → 보유 불가
- Airflow DAG도 **팀별 독립 실행** → 타 팀 데이터를 처리할 경로 자체가 없음

## [Slide 7] 격리 보장 메커니즘 — "필터 누락도 안전한 이유"

### 기존 방식 vs 제안 방식 비교

| 시나리오 | 기존 (메타데이터 필터) | 제안 (인프라 격리) |
|---------|----------------------|------------------|
| 정상 검색 | 팀 A 문서만 반환 | 팀 A 문서만 반환 |
| 필터 버그/누락 | **팀 B 기밀 문서 노출** | **영향 없음 (다른 Collection)** |
| 권한 탈취 시도 | 단일 DB 전체 접근 가능 | 해당 팀 Collection만 접근 가능 |
| 보안팀 감사 | "필터 의존" → 통과 어려움 | "인프라 분리" → 통과 용이 |

### 격리 경계 3중 구조

```
[MinIO 버킷 격리]  →  [DAG 실행 격리]  →  [Collection 격리]
  스토리지 단계           파이프라인 단계         검색 단계
  (저장부터 분리)         (처리도 분리)          (조회도 분리)
```

## [Slide 9] 구현 단계

### Phase 1 — 플랫폼 기반 구축
- agentMarket: 팀 RAG 자원 신청·승인 UI
- HPP: Airflow DAG / MinIO Bucket / Milvus Collection 자동 프로비저닝
- 보안 인증: JWT 발급·검증, Gateway Key 연동

### Phase 2 — 파이프라인 자동화
- S3 이벤트 → Airflow DAG 자동 트리거
- 임베딩 모델 통합 (팀별 모델 선택 옵션)
- 문서 처리 상태 모니터링 (업로드 → 임베딩 → VectorDB 등록)

### Phase 3 — RAG 검색 서비스 연동
- agentChat에서 팀 소속 Collection만 검색
- A2A 협업 시 명시적 허용된 Collection 간 검색
- 검색 감사 로그 (어떤 팀이 어느 Collection 조회했는지 추적)

## [Slide 10] 정리 — 3문장 핵심 메시지

> 1. **인프라 레벨 격리**로 "필터 누락 한 번에 전체 노출"되는 구조적 위험을 제거했습니다.
>
> 2. **플랫폼 자동 프로비저닝**으로 팀이 RAG를 도입하는 데 걸리던 수 주를 수 분으로 단축합니다.
>
> 3. **MinIO → Airflow → Milvus 3중 격리**로 보안팀이 요구하는 "팀 간 문서 접근 불가" 조건을 기술적으로 증명 가능합니다.
