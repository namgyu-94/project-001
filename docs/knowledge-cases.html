Case A — Agent 등록 신청 시 Knowledge 활성화 여부를 결정

Developer가 Agent를 마켓에 등록 신청할 때 Step 1에서 Knowledge 기능 토글과 DAG ID를 함께 입력합니다.
이후 사용자가 문서를 업로드하면 플랫폼 S3에 임시 보관되고, Developer가 doc-approve에서 승인·반려만 하면 됩니다.
DAG은 이미 Agent에 연결되어 있으므로 승인 시 플랫폼이 자동으로 트리거합니다.

장단점

✔ 장점

  • Agent 단위로 DAG이 고정 → 운영 일관성
  • 승인 시 추가 입력 없음 → 승인 프로세스 단순
  • Admin이 Agent 등록 시 DAG 검토 가능
  • DAG 변경 시 Agent 관리 화면에서 중앙 관리

✖ 단점

  • Agent에 DAG을 고정하므로 문서마다 다른 DAG 사용 불가
  • DAG 변경 시 Admin 재승인 필요 여부 별도 결정 필요
  • Knowledge 미사용 Agent는 신청 시 토글 OFF 필요 (UX 추가)

Case B — 문서 승인 시 Developer가 DAG ID를 직접 입력

✔ 장점

  • Agent 등록 폼 변경이 거의 없음 → 기존 신청 프로세스 영향 최소

  • Admin이 어떤 DAG이 사용되는지 사전 파악 불가
